# shadcn/ui Registry Reference

Registry JSON schema, CLI architecture, dependency resolution, namespaced registries, and the copy-paste distribution model. Synthesized from shadcn/ui source and registry specification.

---

## The Copy-Paste Distribution Model

shadcn/ui components live in YOUR codebase, not in `node_modules`. The CLI copies source files directly into your project.

**Why this matters:**
- Full customization: edit the component, it stays edited
- No version lock-in: you own the code
- No upstream dependency on component internals
- Framework-specific: components use your Tailwind config, your theme tokens
- No black-box behavior: read the source, understand what it does

**Trade-off:** Updates require re-running `add` and manually merging changes. Not suitable when you need guaranteed upstream patches.

```
node_modules/shadcn-ui/     ← Does NOT exist
src/components/ui/          ← Components live HERE (your code)
  button.tsx
  dialog.tsx
  card.tsx
```

## `components.json` — Project Configuration

Generated by `shadcn init`. Describes your project's conventions so the CLI places files correctly.

```json
{
  "$schema": "https://ui.shadcn.com/schema.json",
  "style": "new-york",           // "default" | "new-york" — visual style variant
  "rsc": true,                   // React Server Components support
  "tsx": true,                   // TypeScript + JSX
  "tailwind": {
    "config": "tailwind.config.ts",
    "css": "app/globals.css",    // where CSS variables are injected
    "baseColor": "zinc",         // base palette: slate | gray | zinc | neutral | stone
    "cssVariables": true,        // use CSS vars (recommended) vs inline Tailwind classes
    "prefix": ""                 // optional Tailwind prefix (e.g. "tw-")
  },
  "aliases": {
    "components": "@/components",
    "utils": "@/lib/utils",
    "ui": "@/components/ui",
    "lib": "@/lib",
    "hooks": "@/hooks"
  },
  "iconLibrary": "lucide"        // "lucide" | "radix"
}
```

**Style variants:**
- `default`: slightly looser spacing, rounded corners, subtle shadows
- `new-york`: tighter, sharper, higher contrast — the more polished of the two

## Registry JSON Schema

The `registry.json` file at a registry URL describes all available components:

```json
{
  "$schema": "https://ui.shadcn.com/schema/registry.json",
  "name": "my-registry",
  "homepage": "https://my-ui.dev",
  "items": [
    {
      "name": "button",
      "type": "registry:ui",         // see types below
      "title": "Button",
      "description": "A polymorphic button with loading state.",
      "dependencies": [              // npm packages required
        "@radix-ui/react-slot"
      ],
      "devDependencies": [],
      "registryDependencies": [      // other registry items required
        "utils"
      ],
      "files": [
        {
          "path": "registry/ui/button.tsx",
          "type": "registry:ui",
          "target": "components/ui/button.tsx"  // where CLI places the file
        }
      ],
      "tailwind": {
        "config": {
          "theme": {
            "extend": {
              "keyframes": {
                "spin": { "to": { "transform": "rotate(360deg)" } }
              }
            }
          }
        }
      },
      "cssVars": {
        "light": { "--button-radius": "0.375rem" },
        "dark":  { "--button-radius": "0.375rem" }
      },
      "meta": {
        "importSpecifier": "@/components/ui/button",
        "moduleSpecifier": "my-registry/button"
      }
    }
  ]
}
```

**Registry item types:**
| Type | Description |
|------|-------------|
| `registry:ui` | UI component (goes to `components/ui/`) |
| `registry:component` | Page/app-level component |
| `registry:example` | Usage example (not copied to project) |
| `registry:block` | Multi-file block (e.g., auth form) |
| `registry:hook` | Custom React hook |
| `registry:lib` | Utility library |
| `registry:page` | Full page component |
| `registry:theme` | CSS variable theme |
| `registry:style` | Style definition |
| `registry:file` | Arbitrary file |

## CLI Architecture

```bash
# Initialize project — generates components.json, installs deps, injects CSS vars
npx shadcn@latest init

# Add a component — resolves deps, copies files
npx shadcn@latest add button
npx shadcn@latest add dialog card badge  # multiple at once

# Add from custom registry
npx shadcn@latest add https://my-ui.dev/r/fancy-button.json

# Diff existing component vs registry (what changed upstream?)
npx shadcn@latest diff button

# Build registry (for publishing your own)
npx shadcn@latest build

# Migrate from older shadcn versions
npx shadcn@latest migrate
```

**`add` command flow:**
1. Fetch component JSON from registry URL
2. Resolve `registryDependencies` recursively
3. Install npm `dependencies` via package manager
4. Copy `files` to target paths (prompts before overwriting)
5. Merge Tailwind config extensions
6. Inject CSS variable definitions into `globals.css`

## Dependency Resolution

```
Button
  └── registryDependencies: ["utils"]
       └── files: [lib/utils.ts]  →  copied to @/lib/utils.ts

Dialog
  └── registryDependencies: ["button", "utils"]
  └── dependencies: ["@radix-ui/react-dialog", "lucide-react"]
       └── Button resolved first
       └── npm packages installed
```

The CLI builds a dependency graph and installs in topological order. Circular dependencies are detected and rejected.

## Namespaced Registries

```bash
# Official namespaces
@shadcn   — official shadcn/ui registry
@v0       — components built with v0.dev

# Custom registry usage
npx shadcn@latest add @my-org/data-table

# Registry URL format
https://my-registry.dev/r/[component-name].json

# components.json can specify a default registry
{
  "registries": {
    "my-org": {
      "url": "https://my-registry.dev/r",
      "style": "new-york"
    }
  }
}
```

## OKLCH Color System in shadcn

shadcn v2 uses OKLCH for all color tokens. CSS variables defined in `globals.css`:

```css
@layer base {
  :root {
    --background: oklch(1 0 0);
    --foreground: oklch(0.145 0 0);
    --card: oklch(1 0 0);
    --card-foreground: oklch(0.145 0 0);
    --primary: oklch(0.205 0 0);
    --primary-foreground: oklch(0.985 0 0);
    --secondary: oklch(0.97 0 0);
    --secondary-foreground: oklch(0.205 0 0);
    --muted: oklch(0.97 0 0);
    --muted-foreground: oklch(0.556 0 0);
    --accent: oklch(0.97 0 0);
    --accent-foreground: oklch(0.205 0 0);
    --destructive: oklch(0.577 0.245 27.325);
    --border: oklch(0.922 0 0);
    --input: oklch(0.922 0 0);
    --ring: oklch(0.708 0 0);
    --radius: 0.625rem;
    --sidebar: oklch(0.985 0 0);
  }

  .dark {
    --background: oklch(0.145 0 0);
    --foreground: oklch(0.985 0 0);
    --primary: oklch(0.985 0 0);
    --primary-foreground: oklch(0.205 0 0);
    --destructive: oklch(0.704 0.191 22.216);
    /* ... */
  }
}
```

Tailwind `tailwind.config.ts` maps these vars to utility classes:
```ts
theme: {
  extend: {
    colors: {
      background: 'hsl(var(--background))',  // legacy format
      // or modern:
      background: 'oklch(var(--background))',
      primary: {
        DEFAULT: 'oklch(var(--primary))',
        foreground: 'oklch(var(--primary-foreground))',
      },
    },
    borderRadius: {
      lg: 'var(--radius)',
      md: 'calc(var(--radius) - 2px)',
      sm: 'calc(var(--radius) - 4px)',
    },
  },
}
```

## Building a Custom Registry

```
my-registry/
  registry.json          ← component index
  registry/
    ui/
      fancy-button.tsx   ← source files
      data-table.tsx
  public/
    r/
      fancy-button.json  ← per-component JSON (generated by `shadcn build`)
      data-table.json
```

```bash
# Generate public/r/*.json from registry/ source
npx shadcn@latest build

# Serve publicly — components.json references your URL
# npx shadcn add https://your-domain.dev/r/fancy-button.json
```

Per-component JSON (`public/r/fancy-button.json`) is what the CLI fetches. It contains pre-resolved file contents, so consumers don't need access to your source repo.

## `cn()` Utility

Every shadcn project includes this utility. Required by all components:

```ts
// lib/utils.ts — generated by shadcn init
import { type ClassValue, clsx } from 'clsx'
import { twMerge } from 'tailwind-merge'

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}
```

`clsx` handles conditional class logic. `twMerge` resolves Tailwind conflicts (last class wins, intelligently):

```tsx
// Without twMerge: "p-2 p-4" — both applied, browser picks last
// With twMerge:    "p-4" — conflict resolved correctly

cn('p-2', condition && 'p-4')      // → "p-4" if condition
cn('text-red-500', 'text-blue-500') // → "text-blue-500" (conflict resolved)
cn('px-2 py-1', 'p-3')             // → "p-3" (shorthand overrides longhand)
```

## Block System

Blocks are multi-file compositions (e.g., a full authentication form):

```bash
npx shadcn@latest add login-01   # adds a complete login page block
```

Blocks include:
- Page component
- Sub-components used only by that block
- Form validation schemas
- Server actions (if applicable)

Registry JSON for a block:
```json
{
  "name": "login-01",
  "type": "registry:block",
  "files": [
    { "path": "registry/blocks/login-01/page.tsx", "type": "registry:page" },
    { "path": "registry/blocks/login-01/components/login-form.tsx", "type": "registry:component" }
  ],
  "registryDependencies": ["button", "card", "input", "label"]
}
```
